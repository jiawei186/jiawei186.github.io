Concurrency-control

Posted on 2021-03-15 In Database

Concurrency-control

1 事务执行方式

多用户数据库系统
允许多个用户同时使用同一个数据库的数据库系统, 被称为多用户数据库系统

事务串行执行方式
在一个事务执行完成之后, 再执行下一个事务

事务交叉并发执行方式, interleaved concurrency
将一个事务分成多个部分, 执行了一个事务的一部分之后, 立刻执行另一个事务的一部分, 循环往复

交叉并发的意义
一个事务往往需要多个系统资源, 如CPU, 存取数据库, I/O, 通信等
如果一个事务执行到底, 会出现频繁调用释放系统资源这一现象
如果采用交叉并发执行方式, 就可以在同一个系统资源中处理多个事务的要使用到这个系统资源的部分, 不需要过多的调用释放

事务同时并发执行方式, simultaneous concurrency
每个处理机只能处理一个事务, 多个处理机能分别处理一个事务, 真正实现多个事务的同时运行

2 并发控制

事务是并发控制的基本单位

并发控制, 并发控制机制
并发控制, 即用正确的方式调度并发操作, 使一个用户事务的执行不受其他事务的干扰

并发控制的意义
事务交叉并发执行方式中, 可能出现多个事务先后存取同一数据的情况
这种情况可能会存取不正确的数据, 破坏事务的一致性和数据库的一致性
并发控制, 避免了数据的不一致性, 保证了事务的隔离性和一致性

常见的并发控制技术
封锁, locking
时间戳, timestamp
乐观控制法, optimistic scheduler
多版本并发控制, multi-version concurrency control, MVCC

本文只讲封锁

2.1 数据不一致性

丢失修改, lost update
两个事务T1和T2读入同一数据并修改
T2提交的结果破坏了T1提交的结果
导致T1的修改被丢失

并发操作带来数据不一致的例子
a读取机票余额Y, Y=16
b读取机票余额Y, Y=16
a买一张机票, Y=15被写回数据库
b买一张机票, Y=15被写回数据库

上面的例子中, 明明卖出了两张机票, 但机票余额只减少了1

读"脏"数据, dirty read
事务T1修改某一数据并将其写回磁盘
事务T2读取同一数据后, T1由于某些原因而被撤销
T2读取的数据就和数据库中的数据不一致了
T2读取的数据被称为脏数据

不可重复读, non-repeatable read
事务T1读取数据后, 事务T2对其进行了修改
当事务T1再次读该数据时, 无法再现前一次的读取结果

不可重复读又分为三种情况
T2对记录进行了修改
T2删除了部分记录
T2插入了新记录

后2种情况被称为幻影现象

3 封锁

封锁
事务T在对某个数据对象进行操作之前
先向系统发出请求, T对其加锁
加锁后, 事务T就对该数据对象有了一定的控制
在事务T释放锁之前, 其他事务不能更新此数据对象

封锁分为2类 解释
排他锁, 写锁, exclusive locks 事务T对数据对象A上锁X, 则只允许T读取和修改A, 其他事务不能对A加任何类型的锁, 其他事务申请加锁之后会一直等待, 直到T释放A上的锁为止
共享锁, 读锁, share locks 事务T对数据对象A上锁S, 则T可以读A但不能修改A, 其他事务只能再对A加S锁而不能加X锁, 直到T释放A上的锁为止

相容矩阵
最左边一列表示事务T1已经获得的数据对象上的锁的类型, "-"表示没有加锁
最上边一行表示事务T2对同一数据对象发出的封锁请求
T2的封锁请求能否被满足用矩阵中的Y和N来表示
Y表示T2的封锁请求和T1已持有的锁相容, 封锁请求可以被满足
N表示T2的封锁请求和T1已持有的锁冲突, 封锁请求无法被满足

3.1 封锁协议

封锁协议, locking protocol
在运用封锁时, 需要约定一些规则, 这些规则被称为封锁协议

封锁协议为并发操作的正确调度提供了一定程度的保证

封锁协议可分为3类 解释 作用
一级封锁协议 事务T在修改数据A之前, 必须先对其加X锁, 直到事务结束才释放 防止丢失修改
二级封锁协议 在一级封锁协议的基础上, 事务T在读取数据A之前, 必须先对其加S锁, 读完后即可释放S锁 防止读脏数据
三级封锁协议 在一级封锁协议的基础上, 事务T在读取数据A之前, 必须先对其加S锁, 事务结束后才会释放 防止不可重复读

事务结束包括正常结束COMMIT和非正常结束ROLLBACK

以上3级协议的主要区别在于什么操作需要申请封锁, 以及什么时候释放锁

4 可串行化调度

可串行化调度
执行结果等价于串行调度的调度, 即为可串行化调度

一个给定的并发调度, 当且仅当它是可串行化的, 才认为是它是正确调度

判断一个调度是否是串行调度
最简单的判断方法
就是先看该调度
是不是先完成了一个事务的所有操作
然后再完成另一个事务的所有操作

例如

事务T1 事务T2
读B 读A
A=B+1 B=A+1
写回A 写回B

设A, B的初值均为2
按T1→T2T_{1} \rarr T_{2}T1​→T2​次序, 执行结果为A=3, B=4
按T2→T1T_{2} \rarr T_{1}T2​→T1​次序, 执行结果为B=3, A=4
这两种调度都是串行调度
虽然执行结果不同, 但这两种调度都是正确的调度

所有执行结果与上面两种不同的调度, 都是错误的调度

4.1 判断可串行化调度的另一种方法

冲突操作可分为2类 解释
不同事务对同一数据的读写操作 Ti读x, Tj写x, i!=j
不同事务对同一数据的写写操作 Ti写x, Tj写x, i!=j

其他操作是不冲突操作

调换R~i~(x)和W~j~(x)
调换W~i~(x)和W~j~(x)
都是不被允许的
因为这两个操作都会影响数据库的状态

一个调度Sc在保证了冲突操作的次序不变的情况下
通过交换两个事务的不冲突操作的次序
得到的另一个调度Sc’
如果Sc’是串行调度
则Sc为冲突可串行化的调度

如果一个调度是冲突可串行化的调度
那它一定也是可串行化的调度
即冲突可串行化的调度是可串行化的调度的充分条件
但不是必要条件

例如

Sc1=r1(A)w1(A)r2(A)W2(A)r1(B)w1(B)r2(B)w2(B) 更改
r1(A)w1(A)r2(A)r1(B)w1(B)w2(A)r2(B)w2(B) 将w2(A)与r1(B)w1(B)交换
Sc2=r1(A)w1(A)r1(B)w1(B)r2(A)w2(A)r2(B)w2(B) 将r2(A)与r1(B)w1(B)交换

因为Sc2等价于一个串行调度T1, T2, 所以Sc1为冲突可串行化的调度

例如

有3个事务, T1=W1(Y)W1(X), T2=W2(Y)W2(X), T3=W3(X) 调度
L1=W1(Y)W1(X)W2(Y)W2(X)W3(X) L1是一个串行调度
L2=W1(Y)W2(Y)W2(X)W1(X)W3(X) L2是一个可串行化调度, 因为L2的执行结果与L1相同

L2虽然是可串行化的调度, 但它不是冲突可串行化调度

5 封锁的粒度

封锁粒度, granularity
封锁对象的大小, 被称为封锁粒度

封锁对象可被分为2类 细分
逻辑单元 属性值
属性值的集合
元组
关系
索引项
整个索引
整个数据库
物理单元 数据页
索引页
物理记录
封锁粒度与系统的并发度和并发控制的开销密切相关
封锁的粒度越大 封锁的粒度越小
数据库能封锁的数据单元就越少 数据库能封锁的数据单元就越多
并发度就越小 并发度就越大
系统开销小 系统开销大

例如, 若封锁粒度是数据页
T1需要修改元组L1, 则T1必须对包含L1的整个数据页A加锁
然后, 如果T2需要修改A中的另一个元组L2, 则T2必须等待T1释放锁

例如, 若封锁粒度是元组
事务T需要读取整个表, T就必须对表中的每一个元组加锁

5.1 多粒度封锁, multiple granularity locking

多粒度封锁
在一个系统中同时支持多种封锁粒度供不同的事务选择

多粒度树
多粒度树的根节点表示最大的数据粒度, 一般是整个数据库
叶结点表示最小的数据粒度, 一般是元组

多粒度封锁协议允许多粒度树中的每个结点被独立地加锁
对一个结点加锁意味着这个结点的所有后裔结点也被加以同样类型的锁

加锁可分为2类 解释
显式封锁 应事务的要求直接加到数据对象上的锁
隐式封锁 该数据对象没有被独立加锁, 但其上级结点被加锁所以使得该数据对象被牵连加锁

显式封锁和隐式封锁的效果是一样的

如果事务T要对关系R加X锁
系统必须搜索其上级结点, 关系R本身, R的下级结点
如果其中某一个数据对象已经加了不相容锁, 则T只能等待

这种检查方法的效率低

1 数据库

数据库, Database, DB
数据库是将大量的, 计算机加工而成的, 可以进行高效访问的数据保存起来的数据集合

数据库管理系统, Database Manage System, DBMS
用来管理数据库的计算机系统, 被称为数据库管理系统

关系数据库
关系数据库是数据库的一种
关系数据库采用由行和列组成的二维表来管理数据
关系数据库使用专门的SQL语言来对数据进行操作

字段
二维表的列被称为字段

记录
二维表的行被称为记录

关系数据库必须以记录为单位进行数据读写

单元格
一行和一列交汇的方格称为单元格

关系数据库管理系统, Relational Database Management System, RDBMS
关系数据库的管理系统, 被称为关系数据库管理系统

常见RDBMS 公司
Oracle Database 甲骨文
SQL Server 微软
DB2 IBM
PostgreSQL 开源
MySQL 开源

服务器, 服务端
服务器是指, 用来接收其他程序发出的请求, 并对该请求进行相应处理的程序
服务器也指, 装载有这种程序的计算机或其他设备
RDBMS服务器也是一种服务器
说到RDBMS时, 一般指RDBMS服务器

客户端
客户端是指, 向服务器发出请求的程序
客户端也指, 装载有这种程序的计算机或其他设备
RDBMS客户端也是一种客户端

RDBMS客户端将SQL语句发送给RDBMS服务器
RDBMS再根据该SQL语句来返回所请求的数据, 或者对存储在数据库中的数据进行更新

RDBMS可以和它的客户端安装在同一台计算机上, 也可以安装在不同的计算机上
一个RDBMS可以接受多个客户端的请求
承载客户端功能的多个软件可以是不同的软件

因为RDBMS通常需要处理大量的数据, 所以它一般会被安装在性能比较好的电脑上

标准SQL
国际标准化组织, 即ISO, 制定了标准SQL

4 DML

4.1 查询数据

匹配查询, 查询, query
通过SELECT语句查询并选取出必要数据的过程, 被称为匹配查询

排序指定 解释
ASC ascendent, 升序, 从低到高
DESC descendent, 降序, 从高到低

4.2 插入数据

插入数据时
如果列有NOT NULL约束, 就不能插入NULL数据
如果没有, 就可以插入NULL数据

如果在创建表时, 设定了列约束DEFAULT d-0, d-0就是默认值
插入数据时
可以直接写DEFAULT作为数据, 也可以省略有该约束的列, 在插入数据后sql会自动为该列添加默认值的
如果省略了没有DEFAULT d-0约束的列, 在插入数据后sql会自动为该列添加NULL, 如果该列有NOT NULL约束就会出错

4.3 删除数据

DROP TABLE语句会将表完全删除
DELETE语句会将表中的全部数据删除, 但表本身会留下

4.4 更新数据

更新一个记录, 就相当于先删除该记录, 再重新插入一个新记录

4.5 事务

事务, transaction
事务是需要在同一个处理单元中执行的一系列更新处理的集合

不同的RDBMS有不同的事务策略 解释 例如
自动提交模式 每条SQL语句就是一个事务, 不需要COMMIT或ROLLBACK SQL Server, PostgreSQL, MySQL
非自动提交模型 只有在执行COMMIT或ROLLBACK语句时, 一个事务才正式提交 Oracle

ACID特性
RDBMS的事务都遵循四种特性, 统称为ACID特性

ACID特性 解释
原子性, Atomicity 在事务结束时, 其所包含的处理要么全部执行, 要么完全不执行, 不能只执行一部分
一致性, 完整性, Consistency 在事务中的处理, 要满足以前对数据库所设置好的约束
隔离性, Isolation 不同事务之间互不干扰, 在某个事务结束前, 该事务中的处理对其他事务来说不可见
持久性, 耐久性, Durability 在事务提交或事务回滚之后, 该时间点的数据库中的数据状态会被保存, 能被恢复

日志
将事务的执行记录保存到DBMS之外的存储器中, 该执行记录即被称为日志
在发生数据库管理系统故障之后, 可以利用日志将数据库恢复到故障发生前的状态

5 视图

每当我们使用视图时
RDBMS会临时根据视图临时创建一个临时表
然后再进行后续的操作

视图的优点在于
节省了数据库的存储空间
方便了一些常用语句的调用

5.1 创建更新视图

创建视图时, 不能使用ORDER BY子句

定义该视图时同时满足以下条件, 可以对视图使用INSERT, DELETE, UPDATE等更新语句
SELECT子句中未使用DISTINCT
FROM子句中只有一张表
未使用GROUP BY子句
未使用HAVING子句

有些RDBMS的视图修改, 除了要满足以上条件, 还要添加一些独有的语句来允许对视图进行更新

对视图的更新, 对应的表那里也会进行同样的更新

5.2 使用视图

对视图的使用的格式, 和对表的, 没有多大区别

多重视图
以视图为基础所创建的视图, 被称为多重视图

5.3 删除视图

在有些RDBMS中, 因为多重视图的存在, 必须对视图的删除操作设置参数, 来决定是否删除以它为基础的视图
